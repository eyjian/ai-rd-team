# P5：Resource Points 权重校准结论

> 日期：2026-05-03
> 基于 P1 真实实验数据
> 状态：**v1 校准完成，需在实际项目中继续验证**

## 核心结论

**原头脑风暴中的权重普遍偏低约 4 倍，建议调整如下**：

| 权重项 | 原值 | **新值** | 倍数 | 理由 |
|-------|------|---------|------|------|
| `per_member_spawn` | 10 | **40** | ×4 | 每个成员是独立 LLM 会话，有完整 prompt + 系统上下文 + 首次响应开销，单次 spawn 约 3-5K tokens |
| `per_message` | 1 | **2** | ×2 | 一条消息常触发接收方一轮完整响应（读消息+思考+响应），实际 token ≈ 100-300 |
| `per_broadcast_target` | 1 | **2** | ×2 | 按 message 权重同步调整 |
| `per_minute_runtime` | 2 | **5** | ×2.5 | 即使成员闲置也有心跳/轮询消耗 |
| `per_iteration` | 5 | **15** | ×3 | 一轮完整工作包含文件读写、工具调用，实际 token ≈ 2-5K |

## 档位预算相应调整

| 档位 | 原 max_resource_points | **新值** | 对应实际场景 |
|------|----------------------|---------|------------|
| 🟢 Lite | 30 | **120** | 1-2 成员 + 少量迭代（Bug 修复、小改动） |
| 🟡 Standard | 100 | **400** | 4-5 成员 + 中等复杂度（单模块开发） |
| 🔴 Full | 400 | **1500** | 10+ 成员 + 完整流程（复杂系统） |

## 校准逻辑

**锚定关系**：`100 tokens ≈ 1 Resource Point`（基于 Sonnet 定价，1 RP ≈ ¥0.002）

**P1 实际验证**：
- P1 实验估算消耗 ≈ 30K tokens → 300 RP
- 用**原权重**算出：`3×10 + 16×1 + 6×2 + 3×5 = 73 RP`（偏差 4 倍）
- 用**新权重**算出：`3×40 + 16×2 + 6×5 + 3×15 = 227 RP`（偏差 24%，可接受）

## 额外发现

### 1. Spawn 成本远高于预期

每个成员的 spawn 约等于 **30-50 次普通消息**的成本。这意味着：
- **精简角色**比"压缩消息"更能省成本
- Lite 档位去掉 5 个角色 ≈ 省下 150-250K tokens

### 2. 广播成本随成员数线性放大

广播一条消息给 N 个成员，token 成本 ≈ N+1 倍。这与 CodeBuddy 文档警告一致。
- ai-rd-team 必须严格限制 broadcast 使用
- 首选 P2P，广播仅用于"开工"、"完工"等关键节点

### 3. 闲置成员也有成本

即使成员在等消息，每分钟仍有 50-150 tokens 的心跳消耗。这意味着：
- 长时间闲置的成员应主动 shutdown
- 或设置 `max_idle_minutes` 超时自动退出
