# P3：Web 面板与 Team 通信方案选型

## 验证假设
Web 面板需要**实时观察**运行中 Team 的以下信息：
- 成员状态（工作中/完成/失败）
- 消息流（成员间通信）
- 产出制品（新文件、内容变化）
- 资源消耗（消息数、时长、预估 Resource Points）
- 用户对团队的操作（暂停/介入/发消息）

**核心疑问**：Team 和 Web 面板之间用什么通信方案最合适？

## 四种候选方案

### 方案 A：文件监听（Polling / inotify）

```
Team 成员写 → 文件系统 ← Web 后端监听
```

- **写方**：成员通过 `artifact_recorder` 写 `.ai-rd-team/runtime/` 下的文件
- **读方**：Flask 后端用 `watchdog` / 轮询检测变化
- **推送**：后端用 SSE 或 WebSocket 推给前端

**优点**：
- 解耦彻底（成员不需要知道 Web 存在）
- 简单，无额外依赖
- 天然持久化（文件就是日志）

**缺点**：
- 实时性略差（inotify 有毫秒延迟，轮询更差）
- 高频小文件会引起性能问题

### 方案 B：Socket / 命名管道

```
Team 成员 ──[socket]──> Web 后端 ──[WebSocket]──> 前端
```

- **写方**：成员通过 Adapter 的 socket 客户端发事件
- **读方**：Flask 后端监听 socket 端口

**优点**：
- 实时性好
- 低延迟

**缺点**：
- 耦合紧（成员需要了解 Web 位置）
- 端口冲突、连接管理复杂
- 跨进程调试难

### 方案 C：共享状态文件（JSON / SQLite）

```
Team 成员 ──写──> state.json / runtime.sqlite <──读── Web 后端
```

- 双方读写同一个状态源
- 用文件锁或 SQLite 原子写保证一致性

**优点**：
- 状态集中，断点续跑友好
- SQLite 支持并发查询

**缺点**：
- 需要设计 Schema
- 文件锁竞争
- JSON 并发写风险

### 方案 D：HTTP 回调（Webhook 式）

```
Team 成员 ──curl──> Web 后端 API
```

- 成员有权执行 curl（CodeBuddy 可以）
- 每个关键事件 → POST 给 Web 后端

**优点**：
- 语义清晰（REST 接口）
- 成员 Adapter 可选是否上报

**缺点**：
- 成员需要能执行命令（安全约束场景受限）
- 失败重试逻辑需要自己做

## 对比总结

| 维度 | A 文件监听 | B Socket | C 共享状态 | D HTTP 回调 |
|------|-----------|---------|-----------|------------|
| 实时性 | 中 | 高 | 中 | 高 |
| 实现复杂度 | 低 | 中 | 中 | 中 |
| 持久化 | ✅ 天然 | ❌ 需额外 | ✅ | ❌ 需额外 |
| 耦合度 | 低 | 高 | 中 | 中 |
| 跨平台 | ✅ | 需适配 | ✅ | ✅ |
| 断点续跑 | ✅ | ❌ | ✅ | ❌ |

## 原型阶段实现

为验证核心方案，写两个最小 Demo：

### Demo 1：file-watch-approach.py
用 `watchdog` 监听目录，变化时打印事件。用来验证文件方案的实时性。

### Demo 2：simple-http-callback.py
最简 Flask 服务器，接收成员的 HTTP 事件。用来验证 D 方案的可行性。

> 方案 B 和 C 不做 Demo，通过分析排除。

## 决策原则

第一期 ai-rd-team 的选型应优先满足：
1. ✅ **低耦合**：Team 成员不需要知道 Web 的存在
2. ✅ **持久化**：Web 重启不丢数据，支持断点续跑
3. ✅ **简单**：第一期不需要极致低延迟
4. ✅ **CodeBuddy 友好**：不依赖成员执行命令的能力

## 成功标准
输出 `decision.md`，包含：
- 第一期选型：方案 X（推荐 A 文件监听）
- 理由
- 第二期/未来演进路径
- 文件目录结构约定
