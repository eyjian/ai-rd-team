# 我让 7 个 AI 一起写代码

> 不是"提示词工程"，不是"工作流编排"。
> 是"搭一个数字人团队，让他们自己协作把活干完"。

---

## 一、一个让我反复别扭的瞬间

去年开始用 Cursor、Claude Code、CodeBuddy 这类 AI 编程工具时，我跟很多人一样上头——补全更聪明了、改 bug 更快了、写注释像模像样了。

但写到第三个、第四个完整的后端项目时，那种**别扭感**慢慢就出来了。

我写一段需求："做一个博客后端，用 Go + Kratos，要有用户、文章、评论三个模块。"

工具开始干活：先帮我设计 proto 文件，然后开始写 biz 层。但心里清楚——它写完 biz 之后，我得提醒它去写 data 层；data 层完了，我得说"再帮我装配 wire"；wire 完了，再来写测试。

每一步它都做得很好，但**整条研发流程的方向盘始终在我手里**。

这种感觉就像雇了一个超级聪明的实习生，每件事都干得漂亮，但我必须**全程在他身边**，告诉他下一步该做什么。

我开始想：**为什么我得当 PM？我只是想要一个博客后端啊。**

---

## 二、不是要更聪明的助理，是要会协作的团队

往上想一层，会发现现在 AI 编程工具基本都在这条赛道：

> **让单个 Agent 更强**——更长上下文、更准确的工具调用、更好的代码生成。

这条路当然不会错。但它解决不了"研发是个团队活"这件事。一个真实的研发流程里：

- 产品经理想清楚要做什么
- 架构师决定怎么做、把任务拆给开发
- 多个开发并行实现，自己协调接口
- 评审和测试在产出过程中提意见
- DevOps 准备部署

这些角色之间有**真实的对话和分工**——不是按一个流程图按部就班走完。比如开发 1 写到一半发现接口设计有问题，会去找架构师讨论；测试发现 bug 不会写邮件，直接 ping 开发；评审想到一个边界情况，会插话说"这里得考虑一下"。

**多 Agent 协作框架**听起来该解决这个问题。但打开 LangGraph、CrewAI、AutoGen 这些项目，发现它们都长一个样：

```
Node A → Node B → Node C → Node D
  ↓        ↓        ↓        ↓
 Agent 1  Agent 2  Agent 3  Agent 4
```

每个 Agent 是一个节点，开发者画一个 DAG 把它们串起来。这是**工作流编排**——本质是把"研发流程"硬编码成一张图。

但研发不是流水线。研发是一群人**根据当下情况自主决定**该跟谁说话、该写什么文件、该不该重新审视架构。把它建模成 DAG，就像把一个团队建模成一条传送带——失真。

> 公允地说，这些框架对"流程清晰、步骤固定"的场景（比如数据 ETL、审批流）很合适。只是研发这件事，流程本身是研发出来的结果，不是前置条件。

我想要的是**第三条路**：

| 路线 | 模式 | 类比 |
|---|---|---|
| 单 Agent | 一个超级实习生 | 需要全程指挥 |
| 工作流编排 | DAG 串起多个 Agent | 流水线工人 |
| **自主驱动团队** | 7 个角色自主协作 | 真实研发团队 |

我开始动手做这个东西。代号 `ai-rd-team`。

## 三、ai-rd-team 是什么

一句话：**给它一句话需求，它派 2-7 个数字人成员（按角色分工）自主协作产出真实代码。**

七个内置角色，对应一个真实研发团队：

```
┌──────────────────────────────────────────┐
│              PM 周立项                    │
│      （拆任务 / 跟进度 / 整合报告）         │
└─────────────┬────────────────────────────┘
              │
    ┌─────────┼─────────┐
    │         │         │
┌───▼──┐ ┌────▼───┐ ┌───▼────┐
│分析师│ │ 架构师  │ │ DevOps │
│沈需求│ │ 陈架构  │ │  钱  │
└──┬───┘ └────┬───┘ └────────┘
   │          │
   │     ┌────┴─────┐
   │     │          │
   │  ┌──▼──┐   ┌───▼──┐
   │  │开发1│   │ 开发2 │  ← 可伸缩，最多 5 人
   │  │林1号│   │ 林2号 │
   │  └──┬──┘   └───┬──┘
   │     │          │
   │   ┌─┴────┬─────┘
   │   │      │
   │ ┌─▼──┐ ┌─▼───┐
   └─►评审│ │ 测试 │  ← 可伸缩，最多 3 人
     │王1号│ │赵1号 │
     └────┘ └─────┘
```

按需求复杂度，框架会建议三个档位：

- **Lite 档**（1-2 人）：写个 CLI 工具、做个简单脚本
- **Standard 档**（4 人）：做个完整后端模块或一组 API
- **Full 档**（7 人）：完整产品级别，含 PM 协调和 DevOps 部署

每个数字人**不是 prompt 链**——它们是 CodeBuddy（或将来的 Trae、Qoder）平台上派生的真实 sub-agent，**各自有独立上下文**，**互相可以发消息**，**自主决定下一步做什么**。

## 四、真实跑一次：让团队做一个 Go + Kratos 的博客后端

光说理念没意思。下面记一次真实的端到端跑通。

**需求**：一句话——"做一个博客后端，Go + Kratos 框架，含用户注册登录、文章 CRUD、评论功能。"

**档位**：Standard（架构师 1 + 开发 2 + 测试 1）

**结果**（M5 真实跑通的数据）：

| 指标 | 值 |
|---|---|
| 总耗时 | **16 分钟** |
| 主 Agent 在线时间 | spawn 阶段 4 分钟 + stop 1 分钟 = **5 分钟**（中间 10 分钟可以走开） |
| 成员数 | 4（architect / developer_1 / developer_2 / tester） |
| 产出文件 | **53 个**（Go 源码 + proto + 测试 + 设计文档） |
| 编译验证 | `go build ./...` ✅ + `go vet ./...` ✅ + biz 单测全过 |
| 可执行二进制 | **27.6 MB** 的 `blog-server` |
| 总成本 | 162 RP（预算 400，使用率 40%） |

**整个过程没有任何"工作流编排"**。没有人写过"先 architect 出 proto，再 dev 写代码"这种规则。框架只是：

1. 派出 4 个数字人，告诉每个人"你叫什么、你的职责、你的队友是谁"
2. 然后 step back，让他们自己来

### 几个让我惊喜的瞬间

**瞬间 1：架构师主动拆分任务**

architect 写完 `spec-design.md` 和接口契约后，**没等指令**，自己 send_message 给 developer_1：

> "你负责 biz + data 层；接口契约见 `data-interfaces.yaml`。bcrypt 密码哈希用 `internal/pkg/password/`。"

然后给 developer_2：

> "你负责 service + server + cmd 装配；wire 注入手写就行（kratos new 生成的 wire_gen 我已经预生成了）。"

这些都不是 framework 教它的。是 architect 角色在 prompt 里被告知"你是架构师、你负责拆分任务、你可以直接给开发者发消息"，然后它**自主**做了一个真实架构师会做的事。

**瞬间 2：开发者主动加测试，没人让他这么做**

developer_1 写 biz 层时，**自己**给每个文件配了 `*_test.go`。在写 `user_test.go` 时遇到困难（mock UserRepo 接口要写一堆样板代码），**自己**新建了 `mock_test.go` 提取共享 mock——这是个非常工程师的决定。

**没有 prompt 让他这么做**。但 `python-best-practices` Skill（虽然写给 Python 的，但风格通用）里有"测试要可读、要 DRY"——他理解了精神，自己想出了 mock 提取。

**瞬间 3：tester 在 stop 来临时已经完成了骨架**

run 跑到 14:30 时我在 stop_run。这时 tester 还没"完成"——但打开他的目录，已经有：

```
tests/integration/
├── main_test.go            # 测试入口
├── app_stub.go             # mock 应用启动
├── app_factory_real.go     # 真实应用启动
├── helpers_test.go         # 共享辅助
├── user_test.go            # 用户用例骨架
├── post_test.go            # 文章用例骨架
├── comment_test.go         # 评论用例骨架
└── e2e_flow_test.go        # 端到端流程骨架
```

8 个文件，骨架都已完整，只是没有跑通的具体实现。这不是"被打断"——这是一个**正在按部就班推进的测试工程师**留下的现场。

## 五、它怎么做到的（三个关键设计）

### 设计 1：文件即汇报，不靠主 Agent 串话

业界多 Agent 框架的常见模式是"消息总线"：每个 Agent 把状态发回中央协调器，由它转发。

ai-rd-team 走了相反的路：**成员都写文件**。

每个数字人都有：

- 一个 `state/members/<name>.yaml` 自报状态（"我在做什么、做到哪了"）
- 一个工作目录（写代码、写文档、写测试）
- 一个 `events.jsonl` 追加事件（"我开始 X 了"、"我完成 Y 了"）

**Web 面板** 通过监听这些文件的变化，实时展示进度——**主 Agent 完全不参与转发**。

```
        ┌─────────────────────────────┐
        │   .ai-rd-team/runtime/      │
        │                              │
        │   state/members/             │
        │   ├── architect.yaml         │
        │   ├── developer_1.yaml       │
        │   ├── developer_2.yaml       │
        │   └── tester.yaml            │
        │                              │
        │   events.jsonl   ← 全员追加 │
        │                              │
        │   manifest.yaml  ← 产出索引 │
        └────┬───────────────┬─────────┘
             │               │
        ┌────▼────┐     ┌────▼─────┐
        │ Web 面板│     │ Engine    │
        │ 看进度  │     │ 看 events │
        └─────────┘     └───────────┘
```

这个设计的好处：

- **进度感来自行为而非自报**：成员说自己 done 不算数，他写出文件才算数
- **断电恢复**：所有状态在文件里，进程重启可以续跑
- **Web 面板和引擎解耦**：面板可以在引擎不在时也展示历史

### 设计 2：Adapter Bridge——让平台变得可插拔

ai-rd-team 自己**不调用 LLM**。它运行在一个**主 Agent 平台**之上（目前是 CodeBuddy，未来要适配 Trae、Qoder 等），通过这个平台派生 sub-agent 成员。

这意味着同一份 ai-rd-team 代码：

- 在 CodeBuddy 里跑，底层是 Claude-Opus-4.7
- 在 Trae 里跑（未来），底层可能是 Doubao-1.6
- 在 Qoder 里跑（未来），底层可能是 GPT-5

**框架不感知模型**。这是"模型无关性"的重要赌注。

实现方式是 `BaseAdapter` 抽象 + `Bridge` 文件协议：

```
┌──────────────────────────────────────┐
│         ai-rd-team Engine            │
│        （Python 进程）                │
└──────────────┬───────────────────────┘
               │ BaseAdapter 接口
               │ create_team / spawn_member
               │ send_message / team_delete
               ▼
┌──────────────────────────────────────┐
│      CodeBuddyAdapter（M1+）         │
│      TraeAdapter（未来）              │
│      QoderAdapter（未来）             │
└──────────────┬───────────────────────┘
               │ 文件 Bridge 协议
               │ runtime/adapter-intents/{uuid}.json
               │ runtime/adapter-results/{uuid}.json
               ▼
┌──────────────────────────────────────┐
│        主 Agent (CodeBuddy IDE)      │
│   按 Skill 指引读 intent → 调工具    │
│   把结果写回 result                  │
└──────────────────────────────────────┘
```

**为什么用文件而不是 socket / RPC**？因为主 Agent 不能反向 import Python 进程的代码，但任何主 Agent 都能读写文件。文件协议是**最低门槛的跨进程通信**。

M5 之后还加了 **AutoBridgeResponder**——把 `_version`、`_probe`、`shutdown_request` 这些**不需要真工具能力**的 intent 自动应答，让主 Agent 真正只在需要它的时候出现。

### 设计 3：角色 Skills——让架构师真的像架构师

每个角色不仅是个名字。它带着：

- **职责描述**（写在 prompt 模板里）
- **Skills 注入**（架构师默认带 `code-review-checklist`，开发者带 `python-best-practices` + `pytest-guide`，等等）
- **memory_scope 记忆访问范围**（架构师能读 `interface-contracts` 和 `key-decisions`，但不能改 `decisions/` 里的 ADR）

这套机制让 prompt 不再是一个 "you are an architect" 的空话，而是一个**带工艺标准的工程角色**：

```
你是陈架构（architect 角色）。

# 你要做什么
- 设计系统架构和模块划分
- 定义接口契约（写到 docs/design/data-interfaces.yaml）
- 输出技术方案文档（写到 docs/design/ARCHITECTURE.md）

# 你的工艺标准（来自 code-review-checklist Skill）
- 接口契约必须先于实现
- 数据流向要画清楚（mermaid 时序图或类图）
- 任何决定都要有可推翻的理由
- 拆任务给开发者时要明确 acceptance criteria

# 你的协作约束
- 你可以直接 send_message 给团队任何成员
- 不要用 broadcast（除非 Full 档）
- 完成后写一份 report-architect.md 给 PM

# 你能读的记忆
- agent.d/tech-stack-selected：用户选的技术栈
- agent.d/interface-contracts：你自己产出的接口
- agent.d/key-decisions：团队共识
```

这就是为什么 architect 会**自己**主动把任务拆给 developer_1 和 developer_2——他的 prompt 里写了"你的协作约束包括 send_message"和"你的职责包括拆任务"。

## 六、它能用了吗 / 怎么试

**当前状态**：alpha 阶段（v0.2.0a1），M1-M5 已完成，M7（产出物落位优化）刚完成代码部分，PyPI 还没发。

但现在已经可以装来玩。完整 4 步，约 5 分钟：

### Step 1：装 Python 包（让 `ai-rd-team` 命令可用）

```bash
git clone https://github.com/eyjian/ai-rd-team.git
cd ai-rd-team
pip install -e ".[dev]"

# 验证
ai-rd-team version    # → ai-rd-team 0.2.0a1（或当前安装的版本号）
```

### Step 2：装 Skill 到 CodeBuddy（让主 Agent 认得 ai-rd-team 指令）

ai-rd-team 是"嵌入 CodeBuddy 的 Python 引擎"，主 Agent 需要两个 Skill 才能识别 `ai-rd-team run` 命令并应答 bridge intent：

- `ai-rd-team-launcher` — 入口引导
- `ai-rd-team-bridge` — 应答 Python 引擎产生的 intent

仓库本身就是一个标准 CodeBuddy marketplace。一行装：

```bash
# 1. 注册 marketplace（CodeBuddy CLI 会自动 clone）
codebuddy plugin marketplace add https://github.com/eyjian/ai-rd-team.git

# 2. 安装 plugin
codebuddy plugin install ai-rd-team@ai-rd-team

# 3. 重启 CodeBuddy IDE（Skill 启动时才扫描）
```

重启后，CodeBuddy 插件面板能看到 `ai-rd-team`，对话里输入"有哪些可用的 skill"应该能看到 `ai-rd-team-launcher` 和 `ai-rd-team-bridge`。

### Step 3：准备一个工作区（用 example 最快）

```bash
cp -r /path/to/ai-rd-team/examples/01-smart-bookmark ~/demo
cd ~/demo
```

仓库里有 3 个 example 可选：

| example | 档位 | 成员 | 预计耗时 |
|---|---|---|---|
| `01-smart-bookmark` | Lite | 1 个开发 | 5-10 min |
| `02-blog-api` | Standard | 4（架构师 + 双开发 + 测试） | 10-15 min |
| `03-todo-mini` | Standard | 4（微信小程序） | 10-15 min |

每个 example 已经预填好 `.ai-rd-team/config.yaml` 和 `memory/agent.d/` 里的背景知识。

### Step 4：在 CodeBuddy 会话里启动团队

**这一步必须在 CodeBuddy IDE 里操作**，不是终端。

1. 用 CodeBuddy 打开 `~/demo` 作为工作区
2. （可选但推荐）另开一个终端跑 `ai-rd-team serve --port 8765 &`，浏览器打开 `http://127.0.0.1:8765` 看 Web 面板
3. 在 CodeBuddy 对话里输入：

   ```
   ai-rd-team run "$(cat REQUIREMENT.md)"
   ```

   或者直接说自然语言：

   ```
   用 ai-rd-team 帮我做这个项目（参考 REQUIREMENT.md）
   ```

4. 主 Agent（已经加载了 `ai-rd-team-launcher` Skill）会：
   - 启动 Python 引擎
   - 派发团队成员
   - **激活 `ai-rd-team-bridge` Skill** 在后台轮询 intent 并调工具
   - 同步进度到 Web 面板

5. 打开 Web 面板就能看到团队工作的实时进度——成员状态、P2P 消息、产出文件、成本累计。

> ⚠️ **如果团队启动后报 `Tool call timed out: op=team_create`**：说明主 Agent 没激活 bridge Skill。检查 `.ai-rd-team/runtime/adapter-intents/` 是否有未应答的 json 文件，提醒主 Agent："请按 ai-rd-team-bridge Skill 处理 intent"。

完整的 7 步细节、踩坑指南、Web 面板使用、产出验证，见仓库的 [`docs/01-getting-started.md`](https://github.com/eyjian/ai-rd-team/blob/main/docs/01-getting-started.md)。

### 我必须诚实说说它的局限性

**这是 alpha**：

1. **只支持 CodeBuddy 平台**——Trae、Qoder 的 Adapter 还没做。这是有意识的选择：先把一个平台跑透，再扩。
2. **真实 E2E 跑过 5 次**——M1 到 M5 每个里程碑都有真实 CodeBuddy + Claude-Opus-4.7 跑通的报告（见 `prototype/M*-real-e2e/REPORT.md`）。但**没在大规模、长时间任务上验证过**。
3. **成员 state 偶尔滞后**——sub-agent 偶尔忘了更新自己的 yaml 状态，但产出文件是真的。
4. **不适合开放式探索**——它做的是"按角色把研发流程跑完"，不适合那种"我也不知道想要啥，先试试看"的工作。

## 七、写在最后：这条路的赌注

ai-rd-team 是一个赌注：**AI 编程的下一站不是更聪明的单 Agent，而是会协作的团队**。

这个赌注未必赢。也许"超级 Claude" 哪天直接接管所有研发任务，不需要团队建模。也许"工作流编排"才是工业界落地的最优解，自主性反而带来不可控。

但我做了快一年才意识到：**研发的核心难度从来不是"代码怎么写"，而是"事情怎么协调"**。把 AI 武装得再好，只要"协调"没下沉，下一个博客后端到来时，还是得 24 小时盯着实习生。

ai-rd-team 想换一种打法——把"协调"也下沉到框架里，让团队自己去解决。

现在它还能做的不多——alpha 阶段、一个平台、几个示例。但每一次跑通，看着架构师自己拆任务、开发者自己加测试、测试自己搭骨架，我都会有那种感觉：**这些不是 prompt chain 在跑、是一群人在做事**。

如果这个感觉有共鸣——

GitHub：[github.com/eyjian/ai-rd-team](https://github.com/eyjian/ai-rd-team)
