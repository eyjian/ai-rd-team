# Code Review Checklist

## 适用场景

对他人（或自己）的代码进行评审时。

## 核心原则

- **先看设计，再看实现**：接口/命名/依赖方向远比语法更重要
- **指出问题，不指责人**：每条意见附上"为什么"和"建议怎么改"
- **区分 must / should / nit**：必改 / 建议 / 吹毛求疵（用标签前缀）

## 评审维度（逐一检查）

### 1. 正确性
- [ ] 边界情况是否处理（空值 / 超长 / 零 / 负数 / 并发）
- [ ] 错误路径是否有日志 + 上报
- [ ] 外部依赖的失败是否能恢复（重试 / 熔断 / 降级）

### 2. 可读性
- [ ] 命名是否准确（变量名能看出含义，不用缩写）
- [ ] 函数长度 ≤ 30 行，单一职责
- [ ] 注释说"为什么"而不是"做什么"

### 3. 可测试性
- [ ] 纯函数多，副作用集中
- [ ] 外部依赖通过参数/构造函数注入（便于 mock）
- [ ] 关键路径有单测覆盖

### 4. 性能
- [ ] 热点循环里是否有冗余 IO / 重复计算
- [ ] 大数据结构是否应该流式处理
- [ ] 数据库查询是否有索引 / N+1 问题

### 5. 安全
- [ ] SQL 拼接 → 参数化查询
- [ ] 用户输入 → 验证 + 转义
- [ ] 密钥 / token 是否从环境变量读
- [ ] 日志是否泄漏敏感信息

### 6. 文档与测试
- [ ] 公共 API 有 docstring
- [ ] 变更涉及的测试已更新
- [ ] CHANGELOG 有记录（若是对外接口）

## 建议的评审语气

✅ 好：
- "这里改成 `X` 会更清晰，因为 ..."
- "这个边界情况可能会 panic，建议 ..."
- "（nit）命名 `tmp` → `pending_result` 更直观"

❌ 避免：
- "这写得真烂"
- "为什么不用 X？"（带火气）
- 只有 "LGTM" 没有验证过

## 禁止

- ❌ 没看懂就 approve
- ❌ 纠结个人风格（除非团队有约定）
- ❌ 要求对方重写不相关的历史代码

## 快速通过条件

三个都满足则可以直接 approve：
1. 所有 must 项已解决
2. 测试通过
3. 代码行覆盖率没有下降 5% 以上
